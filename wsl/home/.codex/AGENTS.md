# Global Agent Instructions

## Repository and Branch Workflow

- Prefer `gh pr checkout` when checking out an existing pull request.
- Use `ghr clone` when cloning repositories temporarily.
  - Use `ghr list` to inspect existing temporary clones.
- Do not force-push unless it is clearly necessary.
  - Prefer `--force-with-lease` over `--force`.
  - Do not force-push just to clean up history unless explicitly asked.
- If the PR branch has conflicts with the base branch, rebase it onto the latest
  base branch and resolve the conflicts.
  - User approval is not required for this.
  - Do not use merge commits for routine conflict resolution unless repo-level
    instructions require it.

## Pull Request Workflow

- Follow the repository's existing commit title convention for both commit
  subjects and PR titles.
  - Prefer conventional commit format, for example `feat(scope): summary`,
    `fix: summary`, `docs(scope): summary`, or `chore(deps): summary`.
  - Choose the type and optional scope from nearby history and the files being
    changed.
- When code changes are made for a GitHub-hosted repository, create a draft pull
  request by default, even if the user did not explicitly ask for a PR.
- Never mark a draft PR as ready for review unless the user explicitly asks.
- Do not add “Generated by AI”, “Generated by Codex”, AI-authorship footers,
  tool signatures, or similar attribution unless repo-level instructions
  explicitly require it.
- Use closing keywords such as `Closes #123`, `Fixes #123`, or `Resolves #123`
  only when the PR fully resolves the issue.
- For GitHub Discussions, leave a comment linking to the PR when the PR resolves
  or materially addresses the discussion, because GitHub issue-closing keywords
  do not close Discussions.
- If a PR only partially addresses an issue or discussion, describe the scope
  clearly and avoid closing keywords.

## Stacked PR Workflow

When the user says to “stack PR on”, “stack on”, “base this on PR #123”, or
similar:

- Treat it as a request to build the new work on top of the referenced PR’s
  branch.
- This is only a local/source-branch stacking workflow.
- The new PR’s target/base branch must still be the repository’s default branch,
  usually `main` or `master`.
- Do not change the new PR’s target/base branch to the referenced PR’s branch.
- Do not use GitHub’s gh-stack preview flow unless the user explicitly says to
  use it.
- In the new PR description, clearly mention that the PR is stacked on the
  referenced PR.
- If useful, include a note such as:
  - `Stacked on #123. Target branch remains the default branch.`
- If the referenced PR changes while working, rebase the new branch on top of
  the updated referenced PR branch as needed.
- If conflicts occur while stacking or rebasing, resolve them without asking for
  user approval.

## Settling a PR

“Settle the PR” means to keep checking the PR until all relevant CI checks and
AI/code reviews have reached a final state.

Do not settle PRs by default because it consumes tokens and time.

Only settle a PR when the user explicitly asks, for example:

- “settle the PR”
- “wait until the PR settles”
- “follow up until everything settles”
- “wait for checks and reviews”

If the user asks to settle a PR at any point in the session, keep settling that
PR after subsequent follow-up work in the same session, such as fix requests,
until the user explicitly overrides that instruction.

When settling a PR:

- Wait until all relevant CI checks are no longer pending, queued, in progress,
  or waiting.
- Wait until AI/code reviews have completed.
- Apply valid review suggestions.
- For ignored suggestions, leave a brief explanation, then resolve the thread.
- Before treating a CI failure as unrelated, inspect the failed GitHub Actions
  logs and compare against the CI status of the base branch and recent runs for
  other branches, if any have failed, to verify whether the failure is systemic
  rather than caused by this PR.
- If CI fails for reasons unrelated to the change, do not fix unrelated failures
  unless asked. Leave a PR comment explaining why the failure appears unrelated.
- If the branch becomes conflicted with the base branch, rebase it onto the
  latest base branch and resolve the conflicts. User approval is not required
  for this.
- Do not mark the PR as ready for review unless the user explicitly asks.

## Tooling and Dependencies

- Use `mise` for temporary tool execution whenever possible.
  - Prefer: `mise x <tool> -- <bin> ...`
  - Do not install tools into `/tmp` or ad-hoc locations unless `mise` cannot
    provide the tool.
- Avoid global or system-level installs unless explicitly requested.
- Do not install tools globally or modify system packages unless explicitly
  asked.

## Cargo and Local Machine Safety

- Cargo execution may be wrapped to restrict parallel builds across sessions.
- Use `cargo` normally and let the wrapper enforce concurrency limits.
- If Cargo is waiting on another build, lock, or wrapper-controlled slot, wait.
- Do not bypass the Cargo wrapper.
- Do not remove Cargo locks or wrapper locks.
- Do not change build directories to work around waiting.
- Do not launch competing builds to work around waiting.
- Do not attempt Cargo concurrency workarounds unless explicitly instructed;
  doing so may overload or crash the laptop.

## General Behavior

- Prefer small, focused changes.
- Avoid unrelated refactors.
- Do not fix unrelated failures unless explicitly asked.
- Preserve existing formatting and style unless the task requires changing it.
- When uncertain about repository-specific conventions, inspect existing files
  and repo-level instructions before acting.
- If you notice bugs, suspicious behavior, or likely defects while reading code,
  report them to the user.
  - Do not fix those issues unless they are directly in scope for the current
    task or the user explicitly asks.
  - Keep the report concise and include the file/path and why it looks
    problematic.
